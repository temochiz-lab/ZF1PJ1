# ハルシネーション対策

対象: `zend-framework-1-crud-master` ナレッジベース

このページでは、今回のナレッジベース作成で行ったハルシネーション対策をまとめる。

ここでいうハルシネーション対策とは、AIがソースにないことを断定したり、似た名前のファイルや古い流用元コードを混同したり、変更影響を過不足なく確認できないまま答えたりするリスクを下げるための工夫を指す。

## 基本方針

このナレッジベースでは、AIの回答を単独で信じるのではなく、次の形で確認できるようにしている。

```text
説明
↓
各ファイル解説
↓
元ソース
↓
DB定義・画面・URL・処理フロー
```

つまり、説明文だけで終わらせず、できるだけ一次情報や、それに近い整理済み情報へ戻れる構造にしている。

## 対策一覧

| 対策 | 内容 | 効果 |
| --- | --- | --- |
| ソースコードへの参照 | 各ファイル解説から元ソースへリンクする | 実装内容をすぐ確認できる |
| 各ファイル解説 | 元コードと同じフォルダ構成で `ファイル名_解説.md` を作る | ファイルの取り違えを防ぐ |
| ファイル名が出た場所へのリンク | 末尾リンク集ではなく、本文中の該当箇所にリンクする | 説明と根拠の対応が分かりやすい |
| URL一覧 | URL、Controller、Action、View、画面を対応づける | 画面と処理の混同を防ぐ |
| プログラムの流れ | 起動からルーティング、Controller、Model、Viewまで図示する | 処理順序の思い込みを減らす |
| DBテーブル一覧 | SQL定義から `tickets`、`users` を整理する | 存在しないカラムやリレーションの捏造を防ぐ |
| ER図 | 外部キーがないことも含めて図示する | 関係があるように見せてしまうリスクを下げる |
| 関連テーブルリンク | PHP解説から `tickets` / `users` へリンクする | DB変更時の確認漏れを減らす |
| 画面再現PNG | 実行環境なしでも画面イメージを確認できる | 画面仕様の想像補完を減らす |
| 未使用・流用元ファイル一覧 | 現行処理で使われていない可能性があるファイルを分ける | 古いコードを現役機能と誤認しにくい |
| 注意点・改善候補 | 現状の事実と、今後直したい点を分ける | 仕様と提案の混同を防ぐ |
| 脆弱性簡易診断 | 危険箇所を専用ページに分離する | セキュリティ上の懸念を通常仕様と混ぜない |
| 変更影響調査ガイド | 変更時に見るべき観点をまとめる | 固定資料だけで断定するリスクを下げる |
| 詳細仕様と変更ガイド | 詳細資料をフォルダ分けする | 基本資料と変更調査用資料を混同しにくい |
| README | GitHub上の入口を用意する | どこから読むべきか迷いにくい |

## 一次情報に戻れる構造

もっとも重要なのは、説明から一次情報へ戻れること。

今回の構造では、たとえば `TicketMapper` の説明を読んでいるとき、次のように辿れる。

```text
TicketMapper.php の説明
↓
TicketMapper.php_解説.md
↓
元ソース application/models/TicketMapper.php
↓
関連テーブル tickets
↓
08_データベース.md
```

この流れにより、AIが「たぶんこうだろう」で補完した内容を、ソースとDB定義で確認し直せる。

## 複数の観点で同じ事実を確認する

ハルシネーションは、1つの観点だけで説明したときに起きやすい。

そのため、このナレッジベースでは、同じ機能を複数の資料から確認できるようにしている。

例:

| 確認したいこと | 見る資料 |
| --- | --- |
| URLからどの処理が動くか | `05_URL一覧.md`、`06_プログラムの流れ.md` |
| 一覧画面がどのViewか | `05_URL一覧.md`、`15_画面項目一覧.md`、各View解説 |
| チケットがどこに保存されるか | `08_データベース.md`、`TicketMapper.php_解説.md`、`TicketDbTable.php_解説.md` |
| フォーム項目とDBカラムの対応 | `08_データベース.md`、`TopicBootstrapForm.php_解説.md`、`15_画面項目一覧.md` |
| CSV出力の元データ | `Csv.php_解説.md`、`TicketMapper.php_解説.md`、`05_URL一覧.md` |

1つの資料だけで判断せず、複数の資料が同じ方向を指しているか確認する。

## 流用元コードを現行仕様と混同しない

このプロジェクトには、現行のチケットCRUDでは使われていない可能性があるファイルがある。

例:

- `IndexController.php`
- `application/forms/----TopicForm.php`
- `DisplayAddress.php`
- `LoggedInUser.php`
- `users` テーブル

これらを単に「存在するから使われている」と判断すると、AIは誤った説明をしやすい。

そのため、`17_未使用・流用元ファイル一覧.md` を作り、現在の主要処理との距離を明示している。

## 事実と推測を分ける

ナレッジベースでは、次のような表現を分ける。

| 種類 | 書き方 |
| --- | --- |
| ソースから確認できる事実 | 「定義されている」「呼び出している」「設定されている」 |
| 推測を含む説明 | 「と思われる」「可能性がある」「確認が必要」 |
| 改善提案 | 「望ましい」「検討する」「実運用では修正が必要」 |

特に、流用元ファイルや未使用の可能性があるファイルでは断定しすぎない。

## 固定ドキュメントだけで答え切らない

このナレッジベースは、すべての変更影響を事前に書き切るものではない。

たとえば、次の質問には都度調査が必要。

```text
tickets テーブルに is_done カラムを追加した場合、
どのファイルを修正するべきか。
```

この場合、固定ページだけで答えるのではなく、CodexでVaultとソース全体を横断調査する。

確認対象:

- `zf1app_db.sql`
- `Ticket.php`
- `TicketMapper.php`
- `TicketDbTable.php`
- `TopicBootstrapForm.php`
- `TicketController.php`
- `ticket/index.phtml`
- CSV出力
- 画面項目一覧
- 各ファイル解説

## 事実ページと変更ガイドを分ける

ハルシネーション対策として、基本資料と変更ガイドを分けたことも重要。

今回の構成では、`01_` から `12_` までを基本資料として扱い、できるだけソースから確認できる事実を中心にまとめている。

一方で、`詳細仕様と変更ガイド/` 配下には、変更影響、改修パターン、調査プロンプトなど、推測や判断が入りやすい資料を置いている。

```text
基本資料
  事実の要約
  URL / DB / 処理の流れ / ログ / エラー

詳細仕様と変更ガイド
  変更時の確認観点
  推測を含む影響範囲
  改修パターン
  調査プロンプト
```

この区分により、読み手もCodexも「どこまでが確認済みの事実で、どこからが変更時の判断か」を区別しやすくなる。

たとえば、`08_データベース.md` に「`tickets` テーブルには `is_done` カラムは存在しない」と書くのは事実。

一方で、`13_変更影響調査ガイド.md` や `18_改修パターン集.md` に「`is_done` を追加するなら Model、Mapper、Form、View、CSVを確認する」と書くのは変更時の調査方針。

この2つを同じページで混ぜすぎると、AIが「まだ存在しない `is_done` が既に実装されている」と誤読するリスクが上がる。

そのため、事実確認用の基本ページと、推測を含む変更ガイドを分けておくことは、ハルシネーションを抑えるうえで効果がある。

## Codexに調査させるときの聞き方

ハルシネーションを抑えるには、質問の仕方も重要。

よい聞き方:

```text
このVaultとソース全体を見て、
「ticketsテーブルに is_done カラムを追加する場合」
修正が必要なファイル、理由、修正内容、確認した根拠を列挙して。
```

ポイント:

- 「Vaultとソース全体を見る」と指定する。
- 「理由」を求める。
- 「根拠」を求める。
- 「分からないものは分からないと書く」と追加してもよい。

避けたい聞き方:

```text
is_done を追加するには何を直せばいい？
```

これだと、AIが一般的なCRUDの知識で補完しやすい。

## GitHub対応も対策になる

GitHubで読める形にしたことも、ハルシネーション対策になる。

理由:

- 通常Markdownリンクで辿れる。
- 変更履歴が残る。
- どの時点の資料か確認できる。
- レビューしやすい。
- Obsidianだけに閉じず、ブラウザでも確認できる。

## 完全体はVault + Codex

静的なMarkdownだけでは、未来の変更質問に完全には答えられない。

このナレッジベースの完成形は、次の組み合わせ。

```text
元ソースコード
+ 育てたObsidian Vault
+ CodexなどのAIエージェント
+ GitHubによる共有・履歴管理
```

Markdownは答えそのものではなく、Codexが正しく調査するための地図でもある。

## まとめ

今回のハルシネーション対策の中心は、次の3つ。

1. 説明から一次情報へ戻れるようにする。
2. 複数の観点で同じ事実を確認できるようにする。
3. 固定資料で断定せず、変更時はVaultとソースを再調査する。
4. 事実の要約ページと、推測を含む変更ガイドを分ける。

この形にしておくことで、AIの回答をそのまま信じるのではなく、根拠に戻って確認しながら使える。
